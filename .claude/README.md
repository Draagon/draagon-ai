# Claude Code Slash Commands for draagon-ai

This directory contains slash commands that implement GitHub Spec Kit methodology for the draagon-ai cognitive AI framework.

## Available Commands

### Core Spec-Driven Development Workflow

| Command | Purpose | Usage Example |
|---------|---------|---------------|
| `/specify` | Transform feature ideas into specifications | `/specify shared working memory for parallel agents` |
| `/plan` | Generate technical implementation plans | `/plan cognitive swarm orchestration` |
| `/tasks` | Break down plans into actionable tasks | `/tasks phase 2 implementation` |
| `/implement` | Execute specific implementation tasks | `/implement TASK-023` |

### Project Foundation & Quality

| Command | Purpose | Usage Example |
|---------|---------|---------------|
| `/constitution` | Generate/update project principles | `/constitution principles` |
| `/clarify` | Refine and improve specifications | `/clarify belief reconciliation edge cases` |
| `/review` | Comprehensive analysis | `/review all` or `/review cognitive` |
| `/test` | Generate and run tests | `/test cognitive beliefs` |

## Workflow Examples

### Adding a New Cognitive Feature

1. **Specify**: Transform idea into comprehensive spec
   ```
   /specify transactive memory system for expertise routing
   ```

2. **Plan**: Generate technical implementation approach
   ```
   /plan transactive memory implementation
   ```

3. **Tasks**: Break down into actionable development work
   ```
   /tasks transactive memory tasks
   ```

4. **Implement**: Execute each task systematically
   ```
   /implement TASK-024
   ```

5. **Test**: Verify cognitive behavior
   ```
   /test cognitive transactive_memory
   ```

### Improving Existing Specifications

1. **Review**: Assess current specification quality
   ```
   /review specs
   ```

2. **Clarify**: Address gaps or ambiguities found
   ```
   /clarify memory promotion rules
   ```

### Validating Cognitive Architecture

1. **Review cognitive**: Check cognitive components
   ```
   /review cognitive
   ```

2. **Test**: Run cognitive behavior tests
   ```
   /test cognitive all
   ```

## Command Features

### Project-Aware Context

All commands understand:
- **Cognitive architecture** (4-layer memory, beliefs, opinions, curiosity)
- **LLM-first principles** (no semantic regex, XML output)
- **Protocol-based design** (Python Protocols for extensibility)
- **Async-first processing** (non-blocking operations)
- **Research foundation** (cognitive psychology, multi-agent research)

### Consistent Patterns

Commands follow established patterns:
- **FR-XXX** functional requirement numbering
- **TASK-XXX** implementation task format
- **Manual commit workflow** (stage but don't commit)
- **.specify/** directory structure
- **Cognitive testing** requirements

### Constitution Enforcement

All commands validate against constitution:
- No semantic regex patterns
- XML format for LLM output
- Protocol-based integrations
- Confidence-based actions
- Belief-based knowledge storage

## Best Practices

### When to Use Each Command

**Use `/specify`** when:
- Adding new cognitive capabilities
- Defining new memory or belief behaviors
- Creating multi-agent coordination features

**Use `/plan`** when:
- Starting implementation of specified features
- Need technical architecture guidance
- Planning cognitive service integrations

**Use `/tasks`** when:
- Ready to break down implementation work
- Need clear development milestones
- Preparing for actual coding phase

**Use `/implement`** when:
- Ready to write code for specific tasks
- Following task-by-task development approach
- Need structured implementation guidance

**Use `/review`** when:
- Assessing cognitive architecture health
- Checking constitution compliance
- Before major development phases

**Use `/clarify`** when:
- Specifications seem ambiguous
- Found inconsistencies between documents
- Need more cognitive detail

**Use `/constitution`** when:
- Updating project principles
- Onboarding new contributors
- Evolving cognitive architecture

**Use `/test`** when:
- Verifying cognitive behavior
- Generating test coverage
- Running benchmark comparisons

## Tips for Effective Usage

1. **Sequential Workflow**: Follow specify → plan → tasks → implement for best results
2. **Regular Reviews**: Use `/review cognitive` to ensure architecture alignment
3. **Constitution First**: Always check constitution before making changes
4. **Cognitive Testing**: Use `/test cognitive` for belief/memory features
5. **Iterative Clarification**: Use `/clarify` whenever specs feel unclear

## Integration with Manual Commit Workflow

All commands respect the project's manual commit preference:
- Commands stage changes with `git add .`
- NO automatic commits are made
- User maintains control over when to commit
- Follows CLAUDE.md configuration guidelines

## Cognitive Architecture Reminders

### Key Principles
- **LLM-First**: Never use regex for semantic understanding
- **XML Output**: All LLM prompts return XML, not JSON
- **4-Layer Memory**: Working → Episodic → Semantic → Metacognitive
- **Beliefs**: User statements become observations, then beliefs
- **Confidence**: Use graduated confidence levels for actions

### Research Basis
- Miller's Law (7±2) for working memory capacity
- Baddeley's Working Memory Model for layer design
- Wegner's Transactive Memory for expertise routing
- ICML 2025 metacognitive learning for reflection
